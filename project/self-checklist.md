# Самопроверка проекта

| # | Критерий | Да/Нет | Где смотреть / комментарий |
|---|---|---|---|
| 1 | Сервис запускается по инструкциям из `project/README.md` и работает | ДА | `README.md`, `src/service.py`, команда `python -m src.service` |
| 2 | Endpoint `/predict` использует реальную модель, а не заглушку | ДА | `src/predict.py`, `artifacts/ticket_classifier.joblib` |
| 3 | Есть EDA/эксперимент с метриками | ДА | `report.md`, `artifacts/metrics.json`, `python -m src.train` |
| 4 | Есть baseline и улучшенная модель, есть сравнение по метрикам | ДА | `report.md`, `src/model.py`, `artifacts/metrics.json` |
| 5 | Код не свален в один ноутбук, есть структура в `src/` | ДА | `src/data.py`, `src/model.py`, `src/train.py`, `src/service.py` |
| 6 | Есть Dockerfile или понятный сценарий развертывания без Docker | ДА | `README.md`, разделы «Установка» и «Запуск» |
| 7 | Есть `.env.example` и нет реальных секретов/паролей | ДА | `configs/.env.example`, синтетические данные |
| 8 | Реализованы логи/наблюдаемость и `/health` | ДА | `src/service.py`, endpoint `/health`, консольные HTTP-логи |
| 9 | В `report.md` обоснован выбор финальной модели | ДА | `report.md`, разделы 4-5 |
| 10 | `README.md` и `report.md` позволяют понять сценарий демонстрации | ДА | `README.md`, `report.md`, раздел «Сценарий демонстрации» |

Итого по самопроверке: 10/10 пунктов выполнены на рабочем уровне. Главная честная оговорка: датасет синтетический, поэтому качество модели стоит рассматривать как демонстрацию пайплайна, а не готовую production-систему.
