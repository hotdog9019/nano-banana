## 2. Постановка задачи

Предметная область - helpdesk/support triage. Потенциальный пользователь сервиса - оператор поддержки или внутренняя система, которой нужно быстро направить обращение в правильную очередь.

ML-задача формулируется как многоклассовая классификация текста. На вход модель получает поле `text`, на выходе возвращает один из пяти классов:

- `account_access` - проблемы входа, пароля, 2FA, блокировки аккаунта.
- `billing` - платежи, счета, возвраты, подписка.
- `technical_issue` - ошибки API, падения, медленная работа, сбои экспорта.
- `how_to` - вопросы «как сделать».
- `feature_request` - предложения новых возможностей.

Целевые метрики: `accuracy` для общей доли верных ответов и `macro-F1`, потому что классы одинаково важны и нужно видеть качество не только на самом частом классе.

## 3. Данные

Используется файл `data/support_tickets.csv` со 100 синтетическими обращениями. Данные специально небольшие, чтобы проект быстро запускался на обычной машине и не требовал внешних скачиваний. В таблице есть колонки:

- `ticket_id` - идентификатор записи.
- `text` - текст обращения.
- `category` - целевой класс.
- `priority` - демонстрационный приоритет обращения.

В датасете смешаны английские и русские формулировки. Перед обучением удаляются пустые значения, текст приводится к строке и очищается от крайних пробелов. Разбиение train/test выполняется стратифицированно с `test_size=0.2` и `random_state=42`.

## 4. Модели и подходы

Baseline - `DummyClassifier(strategy="most_frequent")`. Он нужен как нижняя граница качества: если ML-модель не обгоняет такой baseline, она не имеет практического смысла.

В эксперименте сравниваются несколько простых подходов: `ComplementNB` на word TF-IDF, `LinearSVC` на char TF-IDF и `SGDClassifier(loss="log_loss")` на char TF-IDF. Финальная модель - пайплайн `TfidfVectorizer + SGDClassifier`. Для признаков выбраны символьные n-граммы `char_wb` длиной 3-5. Такой вариант устойчивее на маленьком русско-английском наборе, чем обычные word n-граммы: он ловит части слов, опечатки и похожие формулировки. Классификатор `SGDClassifier(loss="log_loss")` выбран среди двух лучших моделей, потому что при качестве на уровне `LinearSVC` он дополнительно дает вероятности через `predict_proba`; сервис может возвращать не только класс, но и confidence.

Параметры вынесены в `configs/config.yaml`: путь к данным, размер тестовой выборки, random seed, параметры TF-IDF, путь к модели и порт сервиса.

## 5. Экспериментальный протокол и результаты

Эксперимент воспроизводится командой:

```bash
python -m src.train
```

Скрипт обучает baseline, несколько кандидатных моделей и финальную модель на одном и том же split, считает метрики на test и сохраняет результаты в `artifacts/metrics.json`. Там же сохраняется матрица ошибок финальной модели.

| Модель | Accuracy | Macro-F1 | Комментарий |
|---|---:|---:|---|
| Baseline `most_frequent` | 0.200 | 0.067 | Предсказывает один самый частый класс |
| TF-IDF word + ComplementNB | 0.600 | 0.591 | Сильный простой baseline для текста |
| TF-IDF char n-grams + LinearSVC | 0.700 | 0.665 | Лучшее качество, но нет вероятностей |
| TF-IDF char n-grams + SGDClassifier | 0.700 | 0.665 | Финальная модель: качество как у SVC и есть confidence |

Финальная модель выбрана, потому что она заметно превосходит baseline по обеим метрикам, остается простой для объяснения и запуска, а также возвращает вероятности классов. Главный trade-off - качество пока ограничено синтетическими данными и небольшой тестовой выборкой, зато проект полностью воспроизводим и быстро демонстрируется.

## 6. Архитектура решения и сервис

Пайплайн проекта:

```text
CSV data -> validation/split -> TF-IDF -> classifier -> joblib artifact -> HTTP service
```

Основные модули:

- `src/data.py` - загрузка, валидация и train/test split.
- `src/model.py` - baseline, финальная модель, оценка и таблица предсказаний.
- `src/train.py` - точка входа для обучения и сохранения артефактов.
- `src/predict.py` - загрузка модели и единый контракт предсказания.
- `src/service.py` - HTTP API на стандартной библиотеке Python.

Сервис запускается командой:

```bash
python -m src.service
```

Эндпоинты:

- `GET /health` - проверка, что сервис запущен и модель загружена.
- `POST /predict` - принимает JSON `{"text": "..."}` и возвращает категорию, confidence и вероятности классов.

## 7. Наблюдаемость, конфигурация и безопасность

Сервис пишет базовые HTTP-логи в консоль. Конфигурация лежит в `configs/config.yaml`; переменные окружения можно задать по примеру `configs/.env.example`. Реальных секретов, токенов, паролей и персональных данных в проекте нет.

## 8. Ограничения и дальнейшая работа

Ограничения текущей версии:

- датасет синтетический и пока ограничен разнообразием формулировок;
- нет полноценной кросс-валидации;
- сервис минимальный, без Swagger UI и авторизации;
- качество на редких/неожиданных формулировках может быть нестабильным.

Дальнейшее развитие:

- собрать больше обезличенных обращений;
- добавить k-fold или repeated stratified validation;
- сравнить Logistic Regression, Linear SVM, Naive Bayes и компактные transformer-эмбеддинги;
- добавить FastAPI, Swagger UI и мониторинг распределения входящих категорий;
- логировать ошибки и примеры низкой уверенности для доразметки.
